# Отчёт

## 1) Повышение надёжности и отказоустойчивости

Какие обрабатываются сценарии:

- Пустой или невалидный run_id при загрузке модели из MLflow: выбрасывается ModelLoadError
- Ошибки загрузки модели из MLflow (сеть, артефакты, URI): оборачиваются в ошибку уровня сервиса
- Атомарное обновление модели: если загрузка новой модели завершилась неудачей, предыдущая рабочая модель остаётся активной.
- Запрос к /predict при незагруженной модели возвращает 503
- Отсутствие обязательных фичей для активной модели в /predict возвращает 400
- Модель требует фичи, отсутствующие в схеме сервиса, возвращает 400
- Ошибка инференса (predict_proba упал) возвращает 500 без падения сервиса
- Несоответствие типов в теле запроса (например, строка вместо числа) обрабатывается валидацией FastAPI/Pydantic (422)
- Неудачное обновление модели через /updateModel с невалидным run_id возвращает 400
- Непредвиденные внутренние ошибки в /updateModel возвращают 500

## 3) Логируемые метрики (Prometheus)

### Тех метрики сервиса

- `ml_service_http_requests_total{endpoint,method,status_code}`: число запросов по маршруту и http статусу
- `ml_service_http_request_duration_seconds_bucket{endpoint,method,le}` (+sum/count): сквозная латентность запросов
- `ml_service_errors_total{endpoint,error_type}`: категорзированные ошибки сервиса

Дефолтные коллекторы Prometheus клиента дополнительно отдают метрики процесса и рантайма:
- `process_cpu_seconds_total`, `process_resident_memory_bytes` и др

### Метрики входных данных и предобработки

- `ml_service_preprocess_duration_seconds_bucket`: латентность предобработки
- `ml_service_feature_last_value{feature}`: последнее числовое значение каждой входной фичи

### Метрики модели и инференса

- `ml_service_inference_duration_seconds_bucket`: латентность инференса
- `ml_service_prediction_probability_bucket`: распределение вероятностей предсказания
- `ml_service_prediction_class_total{prediction}`: распределение предсказанных классов

### Жизненный цикл модели и отслеживание продовой модельки

- `ml_service_model_updates_total{status}`: успешные/неудачные обновления модели
- `ml_service_model_update_duration_seconds_bucket{status}`: латентность обновления модели по исходу
- `ml_service_active_model_info{run_id,model_type}`: метаданные текущей активной модели
- `ml_service_active_model_features_count`: количество фичей необходимых активной модели

### Перцентили (75, 90, 95, 99 и 99.9%)

Временны́е перцентили вычисляются в PromQL из бакетов гистограмм
Пример:

```promql
histogram_quantile(0.75, sum(rate(ml_service_http_request_duration_seconds_bucket[5m])) by (le))
```

## 7) Мониторинг дрифта через Evidently

### Архитектура

- ml_service/drift.py: потокобезопасный двухоконный аккумулятор (DriftBuffer) + асинхронная корутина drift_monitoring_loop
- После каждого успешного /predict вызывается `DRIFT_BUFFER.add(features, prediction, probability)`
- При успешном /updateModel вызывается `DRIFT_BUFFER.reset()`: оба окна сбрасываются, новая модель начинает формировать baseline с нуля
- Фоновая корутина регистрируется в lifespan через `asyncio.ensure_future(drift_monitoring_loop())`

### Логика DriftBuffer

1. Первые `REFERENCE_SIZE` (по умолчанию 100) записей заполняют эталонное окно (reference/baseline)
2. Последующие записи идут в текущее окно (не более `CURRENT_MAX = 1000`, а старые записи вытесняются первыми)
3. После успешной загрузки отчёта текущее окно продвигается в эталонное (`rotate()`), текущее сбрасывается и формируется скользящий baseline

### Что мониторится

Каждый отчёт содержит прогон `DataDriftPreset()` по DataFrame со столбцами:
- все входные фичи, используемые текущей моделью (числовые и категориальные)
- `prediction` (метка класса 0/1)
- `probability` (вещественное число, уверенность модели)

Evidently долежен автоматически определять типы столбцов и применяет подходящие тесты на дрифт (Колмогоров–Смирнов для числовых, хи-квадрат для категориальных)
